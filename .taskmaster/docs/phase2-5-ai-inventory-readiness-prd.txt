# Phase 2.5 PRD: AI Inventory Readiness

Suggested file:

`.taskmaster/docs/phase2-5-ai-inventory-readiness-prd.txt`

Suggested TaskMaster tag:

`phase2-5-ai-inventory-readiness`

## 1. Phase 2.5 Goal

Phase 2.5 prepares MiniMall for a later Phase 3 AI inventory assistant without
rewriting the Phase 2 Admin Platform PRD or expanding the current Phase 2 admin
UI scope.

The goal is to provide the minimum backend contracts and traceability needed for
AI-assisted inventory operations:

1. Inventory records are traceable across orders, admin adjustments, inbound
   orders, and future AI suggestions.
2. Operations statistics are available through structured admin APIs.
3. Inbound order drafts can be generated by the system.
4. Inventory changes happen only after an administrator confirms the draft.
5. Future AI code can obtain product, order, and inventory data through
   structured APIs instead of database reads.
6. AI never modifies inventory directly, executes SQL, bypasses backend
   services, or calls `/internal/**`.

## 2. Non-Goals

Phase 2.5 does not include:

1. Rewriting the Phase 2 Admin Platform PRD.
2. Adding AI chat UI, AI sidebars, natural-language workflows, or model
   provider integration.
3. Implementing prompt templates, vector search, RAG, or model invocation.
4. Allowing AI to confirm inbound orders or execute inventory changes.
5. Exposing `/internal/**` to browser clients or AI clients.
6. Adding supplier management, procurement approval chains, or a full ERP flow.
7. Building a BI dashboard; only minimal reusable statistics APIs are required.
8. Refunds, shipping, after-sales, promotions, or expanded order lifecycle work.

## 3. Task List

### Task 1: Define the AI Inventory Integration Contract

Goal:

Lock the Phase 2.5 boundary, data flow, API naming, and acceptance rules before
implementation.

Input:

- Phase 2 Admin Platform PRD.
- `docs/phase2-admin-api-contract.md`.
- Phase 2 task supplements 4.1, 6.1, 6.2, 7.1, and 15.1.

Output:

- `docs/phase2-5-ai-inventory-contract.md`.
- A documented flow: structured data query -> suggestion or draft -> admin
  confirmation -> inventory-service change -> inventory record -> audit log.

Acceptance Criteria:

- The contract explicitly says AI must not directly update inventory, execute
  SQL, bypass backend services, or call `/internal/**`.
- The contract lists the structured APIs that Phase 3 AI is allowed to consume.
- The contract defines that confirmed inventory changes must be executed by
  inventory-service and recorded in inventory records and audit logs.
- The contract documents the Phase 2.5 task outputs and their relationship to
  Phase 3.

Phase 2 Dependencies:

- Phase 2 Task 1: Admin API contract.
- Phase 2 Task 15.1: AI assistant boundaries and Phase 2.5 backlog.

Phase 3 Connection:

- Phase 3 uses this document as the AI inventory assistant system boundary.

### Task 2: Enhance Inventory Record Traceability

Goal:

Make inventory records useful for orders, admin adjustments, inbound orders, and
future AI suggestion links.

Input:

- Existing `inventory_records`.
- Phase 2 Task 6 admin inventory APIs.
- Phase 2 Task 4 audit log infrastructure.

Output:

- Inventory records include traceability fields such as `requestId`, `reason`,
  `adminUserId`, `adminUsername`, `sourceType`, and `referenceNo`.
- `sourceType` supports at least `ORDER_DEDUCT`, `ORDER_RELEASE`,
  `ADMIN_INITIALIZE`, `ADMIN_ADJUSTMENT`, and `INBOUND_ORDER`.
- `AI_SUGGESTION` is reserved as a future source or reference value.

Acceptance Criteria:

- All new admin inventory changes write traceable inventory records.
- Existing order deduct/release behavior remains compatible.
- Duplicate `requestId` behavior is deterministic.
- Inventory record query responses expose source, reason, operator, request id,
  and business reference where present.
- Negative stock remains impossible and returns `BusinessException` through
  `ApiResponse`.

Phase 2 Dependencies:

- Phase 2 Task 3: Gateway role headers and admin protection.
- Phase 2 Task 4: Audit log infrastructure.
- Phase 2 Task 6 and 6.1: Admin inventory API and traceability fields.

Phase 3 Connection:

- AI-generated suggestions or inbound drafts can be traced through
  `referenceNo` and `sourceType`.

### Task 3: Add Safety Stock And Low-Stock Query

Goal:

Provide a structured low-stock baseline for future replenishment suggestions.

Input:

- Product identity.
- Current inventory.
- Safety stock or low-stock threshold value.

Output:

- Inventory admin model includes `safetyStock` or `lowStockThreshold`.
- Admin API can set and read the threshold.
- Admin API exposes low-stock products, for example
  `GET /api/admin/inventories/low-stock`.

Acceptance Criteria:

- Admins can set or update the safety stock threshold for inventory.
- Low-stock state is computed by backend logic, not frontend-only logic.
- Low-stock responses include product id, available stock, locked stock, safety
  stock, and stock state.
- Unauthenticated requests return 401 and USER requests return 403.
- Future AI code can obtain low-stock candidates without database access.

Phase 2 Dependencies:

- Phase 2 Task 3: Admin route protection.
- Phase 2 Task 6 and 6.2: Inventory admin API and low-stock contract.
- Phase 2 Task 13.1: Inventory UI displays low-stock data.

Phase 3 Connection:

- AI replenishment starts from this low-stock candidate set.

### Task 4: Add Reusable Operations Statistics APIs

Goal:

Expose minimal read-only statistics for admin UI and future AI analysis.

Input:

- `orders`.
- `order_events`.
- `inventory_records`.
- Product identifiers and names.

Output:

- Product sales aggregation API, for example
  `GET /api/admin/operation-stats/sales-by-product`.
- Inventory trend API, for example
  `GET /api/admin/operation-stats/inventory-trends`.
- Optional hot/slow product behavior through sorting or query parameters.

Acceptance Criteria:

- APIs support time range filters.
- Sales aggregation returns product id, order count, sold quantity, and amount
  totals.
- Inventory trend returns product id, time bucket, inbound quantity, outbound
  quantity, adjustment quantity, and ending stock.
- APIs are read-only and do not mutate inventory.
- APIs are exposed through `/api/admin/**`; callers do not read service
  databases directly.

Phase 2 Dependencies:

- Phase 2 Task 7: Admin order query API.
- Phase 2 Task 7.1: Product sales aggregation.
- Phase 2 Task 6: Inventory record query.

Phase 3 Connection:

- AI uses these APIs as structured inputs for replenishment and inventory-risk
  analysis.

### Task 5: Implement Inbound Order Drafts

Goal:

Create an inbound order draft model that can later be generated from AI
recommendations.

Input:

- Product ids.
- Suggested inbound quantities.
- Draft source.
- Creator identity.

Output:

- `inbound_order`.
- `inbound_order_item`.
- Status enum: `DRAFT`, `CONFIRMED`, `APPLIED`, `CANCELLED`.
- Draft create API, for example `POST /api/admin/inbound-orders/drafts`.
- Draft query APIs, for example `GET /api/admin/inbound-orders` and
  `GET /api/admin/inbound-orders/{inboundNo}`.

Acceptance Criteria:

- A draft can contain multiple product items.
- Creating a draft does not change inventory.
- Draft status is enum-backed and serialized as stable enum names.
- Draft source supports at least `ADMIN_MANUAL` and reserves `AI_SUGGESTION`.
- Admins can view and cancel drafts.
- All responses use `ApiResponse`; lists use `PageResponse`.

Phase 2 Dependencies:

- Phase 2 Task 3: Admin route protection.
- Phase 2 Task 4: Audit log infrastructure.
- Phase 2 Task 5: Product admin API.
- Phase 2 Task 6: Inventory admin API.

Phase 3 Connection:

- AI suggestions can be converted into inbound order drafts instead of inventory
  changes.

### Task 6: Confirm Inbound Orders And Apply Inventory Changes

Goal:

After an administrator confirms an inbound order, inventory-service applies the
stock increase in a transaction and writes records and audit logs.

Input:

- `inboundNo`.
- Admin identity.
- `requestId` idempotency key.

Output:

- Confirm API, for example
  `POST /api/admin/inbound-orders/{inboundNo}/confirm`.
- Increased inventory.
- Inventory records.
- Admin audit logs.

Acceptance Criteria:

- Only `DRAFT` inbound orders can be confirmed.
- Successful confirmation increases inventory and moves the order to `APPLIED`.
- Repeated confirmation cannot increase stock twice.
- Duplicate `requestId` returns a deterministic result.
- If any item fails, the whole transaction rolls back.
- Each inventory record links back to `inboundNo`.
- Audit log records the admin, inbound order number, item summary, and time.

Phase 2 Dependencies:

- Phase 2 Task 4: Audit log infrastructure.
- Phase 2 Task 6: Idempotent inventory admin API.

Phase 2.5 Dependencies:

- Task 2: Inventory record traceability.
- Task 5: Inbound order draft model.

Phase 3 Connection:

- AI remains advisory; only admin confirmation triggers this inventory change.

### Task 7: Reserve AI Operation Suggestion Records

Goal:

Provide a minimal suggestion review model without implementing model calls.

Input:

- Structured data snapshot reference.
- Suggestion type.
- Suggested items and quantities.
- Suggestion reason.
- Source identity.

Output:

- `ai_operation_suggestion`.
- Status enum: `PENDING_REVIEW`, `CONVERTED_TO_DRAFT`, `REJECTED`,
  `APPLIED`.
- Query API, for example `GET /api/admin/ai-suggestions`.
- Reject API, for example
  `POST /api/admin/ai-suggestions/{suggestionNo}/reject`.
- Convert-to-draft API, for example
  `POST /api/admin/ai-suggestions/{suggestionNo}/convert-inbound-draft`.

Acceptance Criteria:

- Suggestion records cannot directly modify inventory.
- Suggestions can be rejected with a reason.
- Suggestions can be converted to inbound order drafts.
- Converted suggestions link to the generated inbound order number.
- Suggestion records keep an input snapshot reference or structured input
  summary for auditability.
- Phase 2.5 does not generate suggestions with AI; it only stores and routes
  suggestion records.

Phase 2.5 Dependencies:

- Task 4: Operations statistics APIs.
- Task 5: Inbound order draft model.

Phase 3 Connection:

- Phase 3 AI writes or proposes records through this model, then uses the same
  admin review flow.

## 5. Phase 3 AI Inventory Assistant Handoff

Phase 3 should reuse Phase 2.5 in this order:

1. Read low-stock candidates from admin inventory APIs.
2. Read sales and inventory trends from operations statistics APIs.
3. Generate replenishment suggestions.
4. Store suggestions as `ai_operation_suggestion`.
5. Let admins review, reject, or convert suggestions into inbound drafts.
6. Let admins confirm inbound orders.
7. Let inventory-service apply stock changes.
8. Trace all changes through inventory records and audit logs.

The hard boundary remains:

- AI does not update inventory directly.
- AI does not execute SQL.
- AI does not call `/internal/**`.
- AI does not bypass inventory-service.
- Every stock change must be traceable to source, operator, business reference,
  and time.
