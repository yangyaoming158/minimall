## 1. Introduction

MiniMall Order 是一个面向后端开发、微服务架构和高并发订单场景的简化电商系统。项目目标不是构建复杂前端页面，而是实现一个可运行、可压测、可部署、可观测，并可用于简历展示的后端微服务项目。

该系统重点覆盖：

```text
微服务拆分
网关鉴权
服务间调用
缓存
消息队列
订单状态流转
库存一致性
幂等控制
限流
压测
可观测性
Docker Compose 部署
```

---

## 2. Product Overview

MiniMall Order 模拟一个简化电商订单系统，用户可以注册登录、浏览商品、创建订单、模拟支付、取消订单。系统内部通过多个微服务协同完成商品校验、库存扣减、订单状态流转、支付事件处理和通知记录。

系统采用 Java 后端微服务架构，核心服务包括：

```text
api-gateway
user-service
product-service
inventory-service
order-service
payment-service
notification-service
common-lib
```

项目第一版优先保证核心链路跑通：

```text
用户登录
商品查询
库存扣减
创建订单
取消订单
模拟支付
支付成功后订单状态更新
RabbitMQ 事件通知
Docker Compose 启动基础依赖
```

后续阶段逐步加入限流、缓存、压测、Prometheus、Grafana、OpenTelemetry、Jaeger / Tempo 等工程化能力。

## 3. Goals and Objectives

### 3.1 Product Goals

| Goal ID | Goal | Description |
|---|---|---|
| G-001 | 构建可运行的微服务订单系统 | 系统应支持用户、商品、库存、订单、支付、通知和网关服务 |
| G-002 | 实现核心下单链路 | 用户可通过网关创建订单，系统完成商品校验、库存扣减和订单生成 |
| G-003 | 保证库存一致性 | 并发下单时库存不能被扣成负数 |
| G-004 | 支持订单状态流转 | 订单状态必须通过状态机校验，不允许任意更新 |
| G-005 | 支持支付事件驱动 | 支付成功后通过 MQ 通知订单服务和通知服务 |
| G-006 | 支持幂等控制 | 创建订单、扣减库存、释放库存、支付回调、MQ 消费都必须幂等 |
| G-007 | 支持压测与可观测性 | 提供 k6 / JMeter 压测脚本，后续支持 Prometheus、Grafana 和链路追踪 |
| G-008 | 支持 Docker Compose 部署 | MySQL、Redis、RabbitMQ 和业务服务可以通过 Docker Compose 启动 |

### 3.2 Success Metrics

| Metric ID | Metric | Target |
|---|---|---|
| M-001 | 项目可编译 | `mvn clean package -DskipTests` 成功 |
| M-002 | 基础依赖可启动 | `docker compose up -d` 可启动 MySQL、Redis、RabbitMQ |
| M-003 | 核心接口可访问 | 用户、商品、库存、订单、支付接口可通过网关访问 |
| M-004 | 库存安全 | 并发扣减后 `available_stock` 不小于 0 |
| M-005 | 幂等有效 | 重复扣减、重复支付、重复 MQ 消费不会造成重复业务影响 |
| M-006 | 文档完整 | README、架构文档、数据库文档、接口文档、部署文档、压测文档齐全 |

## 8. Business Workflows

### 8.1 Create Order Flow

```text
用户请求创建订单
→ api-gateway 校验 JWT
→ order-service 获取用户 ID
→ order-service 调用 product-service 校验商品存在且上架
→ order-service 调用 inventory-service 扣减库存
→ 库存扣减成功
→ order-service 创建订单，状态为 PENDING_PAYMENT
→ 返回订单号、订单状态、总金额、过期时间
```

#### Acceptance Criteria

```text
库存不足不能创建订单
商品不存在不能创建订单
商品下架不能创建订单
重复提交不能创建重复订单
订单创建成功后库存 available_stock 减少，locked_stock 增加
```

---

### 8.2 Payment Success Flow

```text
用户模拟支付
→ payment-service 修改支付单为 SUCCESS
→ payment-service 发送 PaymentSuccessEvent
→ order-service 消费事件
→ order-service 将订单状态改为 PAID
→ notification-service 消费事件并记录通知日志
```

#### Acceptance Criteria

```text
待支付订单可以支付成功
支付成功后订单状态变为 PAID
重复支付不会重复修改订单
重复 MQ 消息不会重复消费
已取消订单收到支付成功事件必须拒绝或记录异常
```

---

### 8.3 Order Timeout Cancellation Flow

```text
订单创建后状态为 PENDING_PAYMENT
→ 定时任务每 30 秒扫描过期订单
→ 如果 expire_at < now 且状态仍为 PENDING_PAYMENT
→ 修改订单状态为 CANCELLED
→ 调用 inventory-service 释放库存
→ 记录取消原因 PAYMENT_TIMEOUT
```

#### Acceptance Criteria

```text
超时未支付订单会自动取消
自动取消后库存会释放
已支付订单不会被取消
重复取消不会重复释放库存
```

## 9. Order State Machine

### 9.1 Order Status

```text
PENDING_PAYMENT：待支付
PAID：已支付
CANCELLED：已取消
CLOSED：已完成 / 已关闭
REFUNDED：已退款，可选
```

### 9.2 Allowed Transitions

```text
PENDING_PAYMENT → PAID
PENDING_PAYMENT → CANCELLED
PAID → CLOSED
PAID → REFUNDED
```

### 9.3 Forbidden Transitions

```text
CANCELLED → PAID
CLOSED → PAID
REFUNDED → PAID
PAID → PENDING_PAYMENT
```

### 9.4 Requirements

| Requirement ID | Requirement | Description | Priority |
|---|---|---|---|
| STATE-001 | 状态枚举 | 订单状态必须使用枚举 | High |
| STATE-002 | 状态机校验 | 所有状态变更必须经过状态机校验 | High |
| STATE-003 | 禁止随意 update | 不允许绕过状态机直接更新订单状态 | High |
| STATE-004 | 非法状态错误 | 非法状态流转应返回业务异常 | High |

## 13. Consistency and Idempotency Requirements

### 13.1 Inventory Consistency

库存扣减必须使用数据库条件更新：

```sql
UPDATE inventory
SET available_stock = available_stock - #{quantity},
    locked_stock = locked_stock + #{quantity},
    version = version + 1
WHERE product_id = #{productId}
  AND available_stock >= #{quantity};
```

影响行数为 1 表示成功，0 表示库存不足或并发竞争失败。

### 13.2 Idempotent Operations

| Requirement ID | Operation | Required Method |
|---|---|---|
| IDEMP-001 | 创建订单 | Redis SETNX / 数据库唯一约束 / 业务状态判断 |
| IDEMP-002 | 扣减库存 | inventory_records 唯一约束 |
| IDEMP-003 | 释放库存 | inventory_records 唯一约束 |
| IDEMP-004 | 支付回调 | payments.order_no unique + 状态判断 |
| IDEMP-005 | MQ 消费 | event_id 唯一约束 |
| IDEMP-006 | 订单取消 | 状态机判断 |

### 13.3 Distributed Transaction Strategy

MVP 不引入 Seata 或复杂分布式事务。

推荐策略：

```text
本地事务
+ MQ 事件
+ 幂等消费
+ 定时补偿
```

## 17. User Stories and Acceptance Criteria

### US-001 用户注册

As a user, I want to register an account so that I can log in and create orders.

Acceptance Criteria:

```text
用户可以通过注册接口创建账号
用户名不能重复
密码不能明文存储
注册成功返回统一 ApiResponse
注册失败返回统一错误响应
```

### US-002 用户登录

As a user, I want to log in and receive a JWT token so that I can access protected APIs.

Acceptance Criteria:

```text
正确用户名和密码可以登录
登录成功返回 token、userId、username
密码错误不能登录
不存在的用户不能登录
```

### US-003 查询商品

As a user, I want to view product list and product detail so that I can choose a product to order.

Acceptance Criteria:

```text
用户可以查询商品列表
用户可以查询商品详情
商品详情第二次查询优先从 Redis 获取
商品下架后不能被用于创建订单
```

### US-004 创建订单

As a logged-in user, I want to create an order for an available product so that I can pay for it later.

Acceptance Criteria:

```text
未登录用户不能创建订单
商品不存在时创建失败
商品下架时创建失败
库存不足时创建失败
库存充足时创建成功
创建成功后订单状态为 PENDING_PAYMENT
创建成功后库存被锁定
重复提交不会创建重复订单
```

### US-005 取消订单

As a user, I want to cancel a pending payment order so that locked inventory can be released.

Acceptance Criteria:

```text
待支付订单可以取消
取消后状态变为 CANCELLED
取消后库存释放
已支付订单不能直接取消
重复取消不会重复释放库存
```

### US-006 模拟支付

As a user, I want to simulate payment success so that the order can become paid.

Acceptance Criteria:

```text
待支付订单可以支付成功
支付成功后 payment 状态为 SUCCESS
支付成功后发送 PaymentSuccessEvent
order-service 消费事件后订单状态变为 PAID
notification-service 消费事件后记录通知日志
重复支付不会重复修改订单
```

### US-007 订单超时取消

As the system, I want to automatically cancel expired pending orders so that locked stock can be released.

Acceptance Criteria:

```text
系统每 30 秒扫描过期未支付订单
过期 PENDING_PAYMENT 订单被取消
库存被释放
已支付订单不会被取消
重复扫描不会造成重复释放
```

### US-008 管理商品状态

As an admin or internal operator, I want to create products and update product status so that order creation can respect product availability.

Acceptance Criteria:

```text
可以创建商品
可以查询商品
可以上下架商品
商品状态变更后删除 Redis 缓存
下架商品不能被下单
```

### US-009 压测订单接口

As a developer, I want to run pressure tests so that I can evaluate order creation performance.

Acceptance Criteria:

```text
提供 k6 或 JMeter 脚本
压测覆盖商品详情查询和创建订单接口
压测报告包含 QPS、平均响应时间、P95、错误率
高并发下库存不能为负数
```

### US-010 查看监控和链路

As a developer, I want to view metrics and tracing so that I can understand system performance and request flow.

Acceptance Criteria:

```text
服务暴露 Actuator 指标
Prometheus 可以抓取指标
Grafana 可以展示 QPS、延迟和错误率
Jaeger / Tempo 可以查看创建订单链路
```

## 18. Development Phases

### Phase 0: Project Initialization

Goal:

```text
创建 Maven 多模块项目骨架。
```

Deliverables:

```text
父工程 mini-mall-order
common-lib
api-gateway
user-service
product-service
inventory-service
order-service
payment-service
notification-service
每个服务提供 /health 接口
docker-compose.yml 包含 MySQL、Redis、RabbitMQ
README.md 和 docs 目录
```

Acceptance Criteria:

```text
mvn clean package -DskipTests 成功
每个服务能单独启动
docker compose up -d 能启动 MySQL、Redis、RabbitMQ
README 有启动说明
```

### Phase 1: common-lib

Goal:

```text
实现公共模块，统一响应、异常、JWT、用户上下文和基础 DTO。
```

Acceptance Criteria:

```text
所有服务能引用 common-lib
统一响应格式可用
统一异常结构可用
编译无循环依赖
```

### Phase 2: user-service

Goal:

```text
实现用户注册、登录、JWT、查询当前用户。
```

Acceptance Criteria:

```text
用户可以注册
密码不明文存储
用户可以登录并获得 token
携带 token 可以访问 /api/users/me
```

### Phase 3: product-service

Goal:

```text
实现商品管理和商品详情缓存。
```

Acceptance Criteria:

```text
商品列表可以查询
商品详情可以查询
商品详情第二次查询优先走 Redis
商品下架后不能创建订单
```

### Phase 4: inventory-service

Goal:

```text
实现库存查询、扣减、释放、库存流水和幂等。
```

Acceptance Criteria:

```text
库存不足时扣减失败
并发扣减时库存不能为负数
重复扣减同一订单不会重复减少库存
取消订单后可以释放库存
```

### Phase 5: api-gateway

Goal:

```text
实现统一入口、路由、JWT 鉴权、用户信息透传。
```

Acceptance Criteria:

```text
通过网关可以访问各服务接口
未登录不能访问订单接口
登录后可以访问受保护接口
下游服务能拿到用户 ID
```

### Phase 6: order-service

Goal:

```text
实现核心下单、查询、取消和状态流转。
```

Acceptance Criteria:

```text
正常商品且库存充足时可以创建订单
商品不存在、下架、库存不足时不能创建订单
创建订单后库存减少
取消待支付订单后库存释放
已支付订单不能直接取消
重复提交不会产生异常重复订单
```

### Phase 7: payment-service and RabbitMQ

Goal:

```text
实现支付单、模拟支付成功和支付事件驱动。
```

Acceptance Criteria:

```text
待支付订单可以支付成功
支付成功后订单状态变为 PAID
重复支付不会重复修改订单
重复 MQ 消息不会重复消费
notification-service 能记录通知日志
```

### Phase 8: Order Timeout Cancellation

Goal:

```text
使用 Spring Scheduler 实现订单超时自动取消。
```

Acceptance Criteria:

```text
超时未支付订单自动取消
自动取消后库存释放
已支付订单不会被取消
重复取消不会重复释放库存
```

### Phase 9: Rate Limit, Cache and Pressure Test

Goal:

```text
实现基础限流、防重复提交、缓存和压测脚本。
```

Acceptance Criteria:

```text
高频请求会被限流
重复提交订单会被拦截
缓存命中后数据库查询减少
压测报告包含 QPS、平均响应时间、P95、错误率
```

### Phase 10: Observability and Deployment

Goal:

```text
接入 Actuator、Prometheus、Grafana、OpenTelemetry 和 Jaeger / Tempo。
```

Acceptance Criteria:

```text
Docker Compose 可以启动所有服务和依赖
Prometheus 可以抓取服务指标
Grafana 可以看到服务 QPS、延迟和错误率
Jaeger / Tempo 可以看到创建订单链路
README 能指导新用户启动项目
```

## 20. Constraints for AI Coding Agent

开发时 Codex / Cline / TaskMaster 必须遵守：

```text
1. 每次只实现一个 Phase 或一个明确的 task/subtask。
2. 修改前先输出计划。
3. 不要一次性生成全部项目。
4. 不要引入当前阶段之外的复杂组件。
5. 不要硬编码数据库密码、JWT 密钥或服务地址。
6. 订单、库存、支付、MQ 消费相关逻辑必须考虑幂等。
7. 状态字段必须使用枚举。
8. 订单状态变更必须通过状态机校验。
9. 所有接口使用 ApiResponse。
10. 业务异常使用 BusinessException。
11. 每完成一个阶段后更新 README 和 docs。
12. 修改后运行 mvn clean package -DskipTests。
13. 如果有测试，运行对应测试。
14. 如果同一错误连续出现两次，停止自动修复并输出阻塞原因。
```
