Phase 1 PRD：MiniMall Customer Frontend

文件建议位置：
.taskmaster/docs/phase1-customer-frontend-prd.txt

版本：v1.0
阶段：Phase 1 - Customer Frontend
默认技术栈：Vue 3 + TypeScript + Vite + Element Plus + Pinia + Vue Router + Axios

当前说明：
本 PRD 是已完成的 Phase 1 历史基线，接口列表保留阶段设计上下文。
当前唯一浏览器/API 接口总览以 docs/api-gateway-contract.md 为准。

说明：
如果后续决定使用 React，也可以改为 React + TypeScript + Vite + Ant Design + React Router + Zustand/Redux Toolkit + Axios。
当前 PRD 默认采用 Vue 3 + Element Plus，原因是开发效率高、表单/表格/分页组件成熟，后续管理员后台也可复用。

============================================================
一、背景
============================================================

MiniMall Order 后端 MVP 已完成，并且 Phase 0 已完成 API Contract Polish。当前后端已经提供稳定的 canonical API：

- /api/users/**
- /api/products/**
- /api/inventories/**
- /api/orders/**
- /api/payments/**

前端只允许通过 api-gateway 调用后端。
禁止直接访问 user-service、product-service、inventory-service、order-service、payment-service、notification-service 等微服务端口。
禁止调用 /internal/**。
禁止前端主动发送 X-User-Id 或 X-Username。

Phase 1 的目标是开发一个面向普通用户的 MiniMall 用户商城前台，完整跑通：

注册 -> 登录 -> 浏览商品 -> 查看库存 -> 创建订单 -> 查看订单 -> 模拟支付 -> 订单变为 PAID

该阶段不做管理员后台，不做 RBAC，不做购物车，不做收货地址，不做优惠券，不做真实支付，不做退款/售后。

============================================================
二、产品定位
============================================================

Phase 1 产品定位：MiniMall 用户商城前台。

它不是完整淘宝/京东级电商平台，而是一个用于展示 MiniMall 后端微服务订单链路的轻量级用户前台。

界面目标：

1. 简洁现代。
2. 有基础电商前台观感。
3. 能清楚展示商品、库存、订单、支付状态。
4. 能完整演示后端微服务核心链路。
5. 不追求复杂营销页、复杂推荐、购物车、地址管理、优惠券系统。

演示核心价值：

1. 用户认证：注册、登录、JWT 持久化、当前用户。
2. 商品浏览：商品列表、商品详情。
3. 库存展示：可用库存、锁定库存、库存状态。
4. 订单链路：创建订单、我的订单、订单详情、取消订单。
5. 模拟支付：支付成功后订单状态刷新为 PAID。
6. 网关接入：所有请求通过 api-gateway。
7. 统一错误处理：ApiResponse、401、429、业务错误提示。

============================================================
三、非目标
============================================================

Phase 1 明确不做以下内容：

1. 不开发管理员后台。
2. 不实现 RBAC、角色、权限、管理员登录。
3. 不实现商品创建、编辑、上下架 UI。
4. 不实现库存调整 UI。
5. 不实现订单后台管理。
6. 不实现通知日志后台管理。
7. 不实现购物车。
8. 不实现收货地址。
9. 不实现优惠券、促销、积分。
10. 不实现收藏、推荐、评价。
11. 不实现真实支付渠道。
12. 不实现退款、对账、支付回调。
13. 不直接访问微服务端口。
14. 不调用 /internal/**。
15. 不修改后端业务逻辑，除非前端联调发现明确 bug 且经过确认。
16. 不引入复杂前端微前端架构。
17. 不引入服务端渲染框架。
18. 不做移动端 App，只做 Web 前台，PC 优先，兼顾基础响应式。

============================================================
四、用户角色
============================================================

4.1 游客

游客可以：

1. 访问商品列表。
2. 查看商品详情。
3. 查看商品库存。
4. 进入登录页。
5. 进入注册页。

游客不能：

1. 创建订单。
2. 查看我的订单。
3. 查看订单详情。
4. 模拟支付。
5. 取消订单。

游客点击需要登录的操作时，应跳转到登录页，并带上 redirect 参数，登录成功后回到原页面。

4.2 已登录用户

已登录用户可以：

1. 查看当前用户信息。
2. 浏览商品。
3. 查看库存。
4. 创建订单。
5. 查看我的订单。
6. 查看订单详情。
7. 取消待支付订单。
8. 模拟支付待支付订单。
9. 查看支付结果。

4.3 管理员

Phase 1 不支持管理员角色。
如后续 Phase 2 开发管理员后台，应单独写 Admin Platform PRD，并定义 /api/admin/**、RBAC 和 admin-safe APIs。

============================================================
六、页面详细需求
============================================================

6.1 全局布局

顶部导航栏内容：

登录前：
- MiniMall Logo
- 商品
- 登录
- 注册

登录后：
- MiniMall Logo
- 商品
- 我的订单
- 当前用户名
- 退出登录

要求：

1. 所有页面保持统一顶部导航。
2. 当前路由高亮。
3. 用户登录状态由 auth store 控制。
4. 点击退出登录后清理 token、用户信息，并回到 /products 或 /login。
5. 页面主体使用浅灰背景 + 白色卡片布局。
6. PC 端优先，页面宽度建议使用居中容器。
7. 移动端基础适配即可，不要求完整移动商城体验。

6.2 登录页

路由：/login
调用 API：POST /api/users/login

表单字段：
1. username
2. password

交互要求：

1. 用户名和密码不能为空。
2. 登录按钮支持 loading 状态。
3. 登录成功后保存 token。
4. 登录成功后调用 GET /api/users/me 或使用登录响应中的用户信息初始化当前用户。
5. 登录成功后跳转：如果 URL 有 redirect 参数，跳转 redirect；否则跳转 /products。
6. 登录失败时展示后端返回的 message。
7. 401 或业务错误不要导致页面崩溃。
8. 提供“没有账号？立即注册”入口。

UI 设计方向：简洁登录卡片，可包含 MiniMall 品牌说明区域，不需要复杂背景动画。

6.3 注册页

路由：/register
调用 API：POST /api/users/register

表单字段：
1. username
2. password
3. email
4. phone

交互要求：

1. 用户名不能为空。
2. 密码不能为空，最少长度按后端约束或前端基础校验。
3. email 和 phone 按后端要求提交；若后端允许为空，前端可不强制。
4. 注册按钮支持 loading 状态。
5. 注册成功后提示成功，并跳转 /login。
6. 注册失败时展示后端 message。
7. 提供“已有账号？立即登录”入口。

6.4 商品列表页

路由：/products
调用 API：GET /api/products

可选查询参数：page、size、keyword、status。是否真正支持以后端为准。

最低要求：

1. 展示商品卡片网格。
2. 每个商品卡片展示商品名称、商品描述摘要、商品价格、商品状态、查看详情按钮。
3. 支持分页。
4. 支持 loading 状态。
5. 支持 empty 状态。
6. API 错误时展示错误提示。
7. 点击商品卡片或按钮进入 /products/:productId。

商品状态展示：
1. ON_SHELF：上架中。
2. OFF_SHELF：已下架。

如果商品已下架，可以展示，但应弱化购买入口；商品详情页应禁用下单按钮。

UI 设计方向：商品卡片式布局，PC 端建议 3 到 4 列，卡片保持留白，价格醒目。

6.5 商品详情页

路由：/products/:productId
调用 API：
1. GET /api/products/{productId}
2. GET /api/inventories/{productId}

展示内容：
1. 商品名称。
2. 商品描述。
3. 商品价格。
4. 商品状态。
5. 库存信息：availableStock、lockedStock、stockState。
6. 数量选择器。
7. 立即下单按钮。

交互要求：

1. 页面进入时并行或顺序加载商品详情和库存信息。
2. 商品不存在时展示错误状态。
3. 商品下架时禁用下单按钮。
4. 库存不足时禁用下单按钮。
5. 用户未登录点击下单时跳转 /login，并携带 redirect。
6. 数量最小为 1。
7. 数量不能超过库存或后端允许的最大值。
8. 点击立即下单跳转到 /checkout?productId=<productId>&quantity=<quantity>。

UI 设计方向：左侧商品展示区域，右侧商品信息和购买区域；库存信息用卡片或信息框展示。

6.6 下单确认页

路由：/checkout?productId=xxx&quantity=1
调用 API：
1. GET /api/products/{productId}
2. GET /api/inventories/{productId}
3. POST /api/orders

创建订单请求字段：
1. productId
2. quantity
3. idempotencyKey

idempotencyKey 生成要求：

1. 前端每次进入下单确认页可生成一次 idempotencyKey。
2. 用户点击确认下单时使用该 idempotencyKey。
3. 点击按钮后进入 loading 状态，防止重复点击。
4. 网络失败或超时可复用当前 idempotencyKey 重试。
5. 用户重新进入 checkout 页面则生成新的 idempotencyKey。
6. idempotencyKey 可以用 crypto.randomUUID() 或等价方式生成。

展示内容：商品信息、单价、数量、总金额、库存状态、确认下单按钮、返回商品详情按钮。

交互要求：商品下架或库存不足时禁止确认下单；未登录访问 checkout 时跳转 /login；创建订单成功后跳转 /orders/{orderNo}。

6.7 我的订单页

路由：/orders
调用 API：GET /api/orders/my

展示内容：
1. 订单编号 orderNo。
2. 商品摘要。
3. 订单金额 totalAmount。
4. 订单状态 status。
5. 创建时间。
6. 操作按钮：查看详情、去支付、取消。

最低要求：
1. 登录用户可访问。
2. 未登录自动跳转 /login。
3. 支持分页。
4. 支持 loading、empty、error 状态。
5. 点击查看详情进入 /orders/:orderNo。
6. 点击去支付进入 /payments/:orderNo。
7. 点击取消订单可调用取消接口，成功后刷新列表。

订单状态标签建议：
1. PENDING_PAYMENT：待支付，橙色。
2. PAID：已支付，绿色。
3. CANCELLED：已取消，灰色。
4. CLOSED：已关闭，深灰。
5. REFUNDED：已退款，蓝灰，如后端已有。

UI 设计方向：PC 端建议表格，状态标签醒目，操作按钮不要过多。

6.8 订单详情页

路由：/orders/:orderNo
调用 API：
1. GET /api/orders/{orderNo}
2. GET /api/payments/{orderNo}，如果订单已经支付或需要展示支付信息

展示内容：订单编号、订单状态、商品信息、商品数量、商品单价、总金额、创建时间、过期时间、支付信息。

操作按钮：
1. PENDING_PAYMENT：去支付、取消订单。
2. PAID：查看支付信息。
3. CANCELLED / CLOSED：无主要操作，展示状态说明。

交互要求：未登录访问跳转 /login；订单不存在时展示错误提示；用户无权访问时展示 403 或后端 message；取消订单成功后刷新订单详情；支付成功返回后重新拉取订单详情。

UI 设计方向：多张信息卡片，简化时间线可选：订单创建 -> 等待支付 -> 支付成功。

6.9 模拟支付页

路由：/payments/:orderNo
调用 API：
1. GET /api/orders/{orderNo}
2. POST /api/payments/{orderNo}/pay
3. GET /api/payments/{orderNo}
4. GET /api/orders/{orderNo}，支付后刷新订单状态

展示内容：订单编号、支付金额、当前订单状态、支付状态、模拟支付按钮、支付成功结果。

交互要求：
1. 只有 PENDING_PAYMENT 订单显示模拟支付按钮。
2. 点击模拟支付时按钮 loading。
3. 支付成功后展示成功状态。
4. 支付成功后重新查询订单详情，确认订单状态变为 PAID。
5. 如果后端 MQ 消费存在轻微异步延迟，可以做短轮询：最多 3 到 5 次，每次间隔 500ms 到 1000ms。
6. 超过轮询次数后提示“支付已提交，订单状态稍后刷新”。
7. 明确提示“当前为模拟支付，用于演示订单支付事件链路”。

6.10 403 页面

展示：无权限访问、返回商品列表、重新登录。

6.11 404 页面

展示：页面不存在、返回首页或商品列表。

============================================================
七、API Client 设计
============================================================

必须封装统一 API client，不允许页面组件直接散落 axios 调用。

建议目录：

src/api/http.ts
src/api/user.ts
src/api/product.ts
src/api/inventory.ts
src/api/order.ts
src/api/payment.ts

http.ts 要求：

1. 设置 baseURL。
2. 从环境变量读取 VITE_API_BASE_URL。
3. 默认 baseURL 可为 http://localhost:8080，具体端口以 api-gateway 为准。
4. 请求拦截器自动附加 Authorization: Bearer <token>。
5. 响应拦截器统一处理 ApiResponse。
6. success === true：返回 data。
7. success === false：抛出带 code/message 的业务错误。
8. HTTP 401：清除 token，跳转 /login，提示登录已过期。
9. HTTP 429：提示请求过于频繁，请稍后再试。
10. HTTP 500：展示系统异常提示。
11. 不发送 X-User-Id 或 X-Username。
12. 不调用 /internal/**。

ApiResponse 类型：

interface ApiResponse<T> {
  success: boolean
  code: string
  message: string
  data: T | null
}

分页类型：

interface PageResponse<T> {
  content: T[]
  page: number
  size: number
  totalElements: number
  totalPages: number
}

============================================================
八、状态管理
============================================================

推荐使用 Pinia。

必须实现 authStore：

1. token
2. currentUser
3. isLoggedIn
4. login()
5. register()
6. fetchCurrentUser()
7. logout()

可选 productStore / orderStore。若页面简单，可以页面内请求，不要过度抽象。

localStorage：

1. token 可保存在 localStorage。
2. 刷新页面时从 localStorage 恢复 token。
3. 应用启动后如有 token，调用 /api/users/me 恢复当前用户。
4. token 无效时清理并跳转登录。

九、路由守卫
============================================================

公开路由：
1. /
2. /products
3. /products/:productId
4. /login
5. /register
6. /403
7. /404

需要登录的路由：
1. /checkout
2. /orders
3. /orders/:orderNo
4. /payments/:orderNo

守卫逻辑：

1. 如果目标路由需要登录但没有 token，跳转 /login?redirect=<currentPath>。
2. 如果已登录用户访问 /login 或 /register，可跳转 /products。
3. 不在路由守卫里调用复杂 API，避免无限循环。
4. currentUser 可在应用初始化或页面内补齐。

十三、错误处理要求
============================================================

13.1 统一业务错误

后端返回 ApiResponse success=false 时：展示 message，可以记录 code，不展示原始堆栈，不让页面崩溃。

13.2 401

清理 token，清理 currentUser，跳转 /login，提示登录已过期，请重新登录。

13.3 403

跳转 /403 或展示无权限提示。

13.4 404

商品或订单不存在时展示友好提示；页面路由不存在时展示 404 页面。

13.5 409 或业务冲突

例如库存不足、订单状态不允许取消、重复支付、订单已过期。应展示后端 message，必要时刷新当前数据。

13.6 429

展示“请求过于频繁，请稍后再试”。不重复自动重试。

13.7 网络错误

展示“网络连接异常，请稍后重试”。表单提交失败后恢复按钮状态。

============================================================
十四、关键业务交互要求
============================================================

14.1 下单幂等

1. 下单请求必须包含 idempotencyKey。
2. 点击确认下单后按钮进入 loading。
3. loading 期间禁止再次点击。
4. 重复点击不会发出多次有效请求。
5. 请求成功后跳转订单详情。
6. 请求失败后展示错误，并恢复按钮状态。

14.2 支付后订单状态刷新

1. 支付成功后先展示支付成功。
2. 然后重新请求订单详情。
3. 如果订单状态已经 PAID，展示已支付。
4. 如果订单状态暂未更新，可短轮询。
5. 轮询失败后提示稍后刷新，不应认为支付失败。

14.3 取消订单

1. 只有 PENDING_PAYMENT 状态显示取消按钮。
2. 取消前弹出确认。
3. 取消成功后刷新订单状态。
4. 如果后端返回订单已支付或不可取消，展示 message 并刷新订单。

14.4 登录跳转

1. 未登录访问受保护路由跳转登录。
2. 登录成功后回到 redirect。
3. 如果没有 redirect，回到 /products。
