## 2. Product Goal

Build an LLM-driven AI Product Inventory Operations Assistant for MiniMall
administrators.

Administrators use the admin frontend to ask natural-language inventory
questions, request low-stock and hot-product analysis, generate replenishment
suggestions, review AI suggestions, convert approved suggestions into inbound
order drafts, and then use the existing inbound order confirmation workflow for
formal stock changes.

The assistant is advisory. It may analyze data and create validated suggestions
or inbound drafts, but it must never directly change inventory.

P0 capabilities:

1. Inventory Q&A.
2. Low-stock analysis.
3. Hot-product analysis.
4. Replenishment suggestion generation.
5. AI suggestion review support through the existing suggestion APIs.
6. Inbound order draft generation from AI suggestions.
7. Suggestion status synchronization after inbound order application.
8. Admin AI inventory assistant page.

P1 capabilities:

1. Daily inventory operations report.
2. Suggestion history enhancements.
3. Demo data generator.
4. Inventory trend visualization.
5. Slow-moving product analysis.
6. Deeper inventory change explanation.

## 3. Hard Boundaries

Phase 3 must preserve these rules:

1. AI code must not directly update `inventory`.
2. AI code must not directly insert or mutate `inventory_records`.
3. AI code must not execute SQL.
4. AI code must not bypass application services or call service ports directly.
5. AI-facing and browser-facing calls must enter through `api-gateway`.
6. AI-facing and browser-facing calls must not call `/internal/**`.
7. AI-facing and browser-facing calls must not send or spoof `X-User-Id`,
   `X-Username`, `X-User-Role`, or `X-Internal-Token`.
8. AI APIs must not confirm inbound orders.
9. AI APIs must not call `POST /api/admin/inventories/{productId}/adjust`.
10. AI responses must not claim stock changed before the existing inbound
    confirmation workflow actually applies the change.
11. Model output must be validated before persistence.
12. Every new REST API must return `ApiResponse`.
13. Paginated list responses must use `PageResponse`.
14. Business errors must use `BusinessException`.
15. Status fields must use stable enum names.

Allowed inventory-assistant flow:

```text
Admin frontend
  -> api-gateway /api/admin/**
  -> AI analysis endpoint
  -> structured backend data facade
  -> LLM provider
  -> backend output validation
  -> ai_operation_suggestion PENDING_REVIEW
  -> administrator review
  -> existing convert-inbound-draft API
  -> inbound_order DRAFT
  -> existing administrator inbound confirmation
  -> inventory-service transaction
  -> inventory_records + admin_operation_logs
  -> suggestion APPLIED synchronization
```

## 5. Existing Status Model

Phase 3 must use the current enum names.

AI suggestion statuses:

1. `PENDING_REVIEW`: generated and validated AI suggestion awaiting admin
   decision.
2. `CONVERTED_TO_DRAFT`: admin converted the suggestion into an inbound order
   draft.
3. `REJECTED`: admin rejected the suggestion.
4. `APPLIED`: the linked inbound order was formally applied.

Inbound order statuses:

1. `DRAFT`: reviewable inbound order draft; does not change inventory.
2. `CONFIRMED`: transitional persisted confirmation state used by the existing
   service flow.
3. `APPLIED`: inventory increase was applied by `inventory-service`.
4. `CANCELLED`: draft was cancelled before application.

Do not introduce the alternative suggestion statuses `DRAFT`, `CONFIRMED`, or
`EXECUTED` in Phase 3 tasks. They conflict with the completed Phase 2.5 API and
database contracts.

## 7. AI Data Ownership And Service Placement

P0 implementation should live in `inventory-service`.

Rationale:

1. `inventory-service` owns `inventory`, `inventory_records`, `inbound_order`,
   `inbound_order_item`, `ai_operation_suggestion`, and suggestion conversion.
2. Putting Phase 3 P0 in `inventory-service` avoids a new distributed
   transaction boundary.
3. Existing admin gateway routes already send `/api/admin/ai-suggestions/**`,
   `/api/admin/inbound-orders/**`, and inventory data to `inventory-service`.

Phase 3 must add a gateway route for `/api/admin/ai/**` to the selected owner.
For P0, route it to `inventory-service`.

A separate `ai-assistant-service` is out of scope for P0. It may be considered
later if the assistant expands beyond inventory.

### 9.1 Inventory Ask

Request:

1. `question`: required natural-language question.
2. `productId`: optional product id hint.

Behavior:

1. Run intent detection against a backend whitelist.
2. Reject unsupported intents with a controlled unsupported-intent response.
3. Execute only read tools.
4. Provide the LLM only relevant structured data.
5. Return answer text, detected intent, query time, time range when applicable,
   and structured evidence.

Supported P0 intents:

1. Current inventory lookup.
2. Low-stock list.
3. Specific product inventory status.
4. Recent inventory record explanation when records exist.

### 9.2 Low-Stock Analysis

Behavior:

1. Read low-stock candidates from backend-computed inventory APIs.
2. Use last 7 days as the default sales window.
3. Include current stock, locked stock, available stock, safety stock, sales
   evidence, and recent records when available.
4. Let the LLM produce reason, risk level, limitations, and suggested quantity.
5. Validate every product id, risk level, and suggested quantity before
   returning or persisting results.

Products without enough data must be marked with explicit limitations, not
invented trends.

### 9.3 Hot-Product Analysis

Behavior:

1. Supports `days=7` and `days=30` in the POST request body.
2. Reads sales aggregation from the existing operation-stat API.
3. Joins current inventory evidence through backend services.
4. Returns product id, product name, sold quantity, order count, sales amount,
   current stock, safety stock, risk level, reason, limitations, and time range.

### 9.4 Replenishment Suggestion Generation

Behavior:

1. Read low-stock products.
2. Read 7-day sales stats.
3. Read hot-product stats and optional inventory records.
4. Send structured evidence to the configured LLM provider.
5. Parse JSON output.
6. Validate output.
7. Persist one `ai_operation_suggestion` only after validation passes.
8. Persist items with suggested quantities and risk levels.
9. Set suggestion status to `PENDING_REVIEW`.
10. Do not change inventory.
11. Do not create an inbound order draft automatically.

## 10. Logical Tools

Separate read tools from write actions.

Read tools allowed during Q&A and analysis:

1. `getInventory(productId)`
2. `listLowStockProducts()`
3. `getSalesStats(days)`
4. `getHotProducts(days)`
5. `getInventoryRecords(productId)`
6. `getOperationSummary(days)`

Write actions allowed only from explicit admin-triggered API workflows:

1. `createSuggestionRecord(payload)`
2. `convertSuggestionToInboundDraft(suggestionNo)`

Rules:

1. The LLM may request or recommend a tool, but backend code decides whether to
   execute it.
2. Q&A endpoints must never execute write actions.
3. Suggestion generation may persist only a validated suggestion.
4. Conversion to inbound draft must use the existing
   `/api/admin/ai-suggestions/{suggestionNo}/convert-inbound-draft` behavior.
5. No logical tool may confirm an inbound order.

## 11. Prompt And Output Guardrails

System prompt rules:

1. Use only provided JSON data.
2. Do not invent product, order, sales, inventory, or date facts.
3. If data is missing, state the limitation.
4. Do not claim inventory changed.
5. Do not output SQL.
6. Do not mention internal service endpoints.
7. Return structured JSON when requested.

Backend validation rules:

1. JSON must parse successfully.
2. `analysisType` must map to an enum.
3. `riskLevel` must be one of `LOW`, `MEDIUM`, or `HIGH`.
4. Every `productId` must come from the input snapshot.
5. Numeric fields must be non-negative.
6. Replenishment `suggestedQuantity` must be a positive integer.
7. Output must not claim inventory has already changed.
8. Output must not contain SQL.
9. Output must not cite products, quantities, sales values, stock values, or
   dates absent from the structured input.
10. Invalid output returns a controlled `ApiResponse` error and is not saved as
    a valid suggestion.

Expected model output shape:

```json
{
  "summary": "string",
  "analysisType": "LOW_STOCK",
  "timeRange": {
    "from": "string",
    "to": "string"
  },
  "items": [
    {
      "productId": "string",
      "productName": "string",
      "availableStock": 0,
      "lockedStock": 0,
      "safetyStock": 0,
      "soldQuantityLast7Days": 0,
      "suggestedQuantity": 1,
      "riskLevel": "LOW",
      "reason": "string"
    }
  ],
  "limitations": ["string"]
}
```

## 15. Main Risks And Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| PRD/status drift from Phase 2.5 | Tasks implement conflicting lifecycle names | Keep this PRD aligned with current enums and existing APIs before parsing |
| AI hallucination | Wrong operational advice | Provide only structured data, require JSON output, validate every product, quantity, risk, date, and claim |
| Provider instability | Demo failure | Include `MOCK` provider and controlled provider failure responses |
| Duplicate inbound execution | Inventory over-counting | Reuse existing inbound confirmation idempotency and never let AI confirm inbound orders |
| Frontend/backend contract drift | Integration delay | Add typed admin API clients and keep review actions on existing APIs |
| Overbuilding agents | Schedule failure | Use whitelisted backend tools and deterministic orchestration, not autonomous multi-agent behavior |
| Security boundary regression | Unauthorized stock or data access | Keep gateway ADMIN checks, downstream ADMIN checks, trusted-header stripping, and `/internal/**` blocking tests |
