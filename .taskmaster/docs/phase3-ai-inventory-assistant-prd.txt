## 1. Overall Feasibility

Phase 3 is feasible for a two-week integrated training project only if the
Phase 2.5 readiness work is completed first.

Current repository capabilities:

1. Product, order, inventory, and inventory record base tables already exist.
2. Orders store `product_id`, `product_name`, `quantity`, `total_amount`,
   `status`, and timestamps, so basic 7-day and 30-day sales aggregation is
   possible.
3. Inventory supports current stock and locked stock.
4. Inventory deduct/release operations are already idempotent by order number
   and change type.
5. User roles and JWT role claims have started in Phase 2.

Current gaps:

1. Admin `/api/admin/**` route protection is not complete.
2. No finished admin inventory API for initialization, adjustment, low-stock
   query, or record timeline.
3. Existing `inventory_records` only tracks product, order number, change type,
   quantity, and status. It does not yet track before/after quantities, source,
   operator, request id, or reason.
4. No `admin_operation_logs` implementation is present yet.
5. No `inbound_order`, `inbound_order_item`, or `ai_operation_suggestion`
   tables are present yet.
6. No admin operation statistics APIs exist yet.
7. No independent `admin-frontend` implementation exists yet.

Conclusion:

Phase 3 should not directly implement stock mutation foundations. Those belong
to Phase 2.5. Phase 3 should focus on AI-facing analysis, suggestion generation,
review UI, and integration with the Phase 2.5 business workflow.

## 2. Product Goal

Build an LLM-driven AI Product Inventory Operations Assistant for MiniMall
administrators. Administrators use natural-language instructions in the admin
frontend, and the assistant uses product, order, inventory, inventory record,
and operation statistics data to provide:

1. Inventory Q&A.
2. Low-stock analysis.
3. Hot-product analysis.
4. Replenishment suggestion generation.
5. AI suggestion history and review.
6. Inbound order draft generation from AI suggestions.
7. Admin-reviewed inventory execution through the inventory service.
8. Daily inventory operations report.

The assistant is intelligent, not template-only. The LLM is responsible for
understanding administrator intent, requesting relevant structured data,
performing inventory and sales analysis, and generating operational
recommendations. The backend is responsible for permissions, structured data
access, tool execution, output validation, suggestion persistence, draft
creation, and formal inventory execution after administrator confirmation.

The assistant must remain advisory. It must not directly update inventory.

## 3. Core Principles

1. AI does not directly modify inventory.
2. AI does not directly execute SQL.
3. AI does not bypass backend business services.
4. AI analyzes structured system data and generates analysis, operation
   suggestions, and document drafts.
5. After administrator confirmation, inventory-service executes the formal
   inventory change.
6. Every inventory change must write inventory records and admin operation
   logs.
7. AI responses must be based on structured system data and must not invent
   facts.
8. `/internal/**` remains blocked for browser and AI-facing callers.
9. All admin-facing APIs return `ApiResponse`.
10. Status fields use enums.
11. Backend validates model outputs before saving suggestions or creating
    drafts.

## 4. Phase 2.5 Entry Gate

Phase 3 implementation can start only after the following Phase 2.5 capabilities
exist:

1. Admin authentication and `/api/admin/**` authorization work.
2. Admin product list/detail APIs are available.
3. Admin inventory APIs can query current stock, low-stock products, and record
   timelines.
4. Inventory includes `safetyStock` or `lowStockThreshold`.
5. Inventory records include source, before/after quantity, operator, reason,
   request id, and business reference.
6. Admin operation logs exist.
7. Sales aggregation APIs exist for product-level 7-day and 30-day analysis.
8. Inbound order draft APIs exist.
9. Inbound order confirmation increases inventory transactionally, writes
   inventory records, writes admin operation logs, and prevents duplicate
   confirmation.
10. AI suggestion records exist as a business object, even if Phase 2.5 does
    not yet generate suggestions with a model.

If any entry-gate item is missing, implement it in Phase 2.5 rather than hiding
it inside Phase 3.

## 5. P0 Scope

### 5.1 Inventory Q&A And Query

The assistant supports administrator natural-language questions and maps them to
controlled inventory analysis intents.

1. Query current inventory.
2. Query low-stock products.
3. Query a specific product's inventory status.
4. Explain a product's recent inventory changes when record data exists.

Acceptance criteria:

1. The administrator can ask in natural language, for example "哪些商品库存有风险"
   or "分析 SKU-001 当前库存状态".
2. The backend creates a controlled tool plan and provides the LLM only the
   relevant structured data.
3. Answers cite the queried product, current stock, locked stock, safety stock,
   sales window, inventory records, and stock state when available.
4. Unsupported questions return a controlled unsupported-intent response, not a
   fabricated answer.
5. Answers include the data time range or query time.
6. The assistant uses only structured backend APIs and admin-safe logical tools.

### 5.2 Low-Stock Analysis

The LLM analyzes low-stock products using backend-provided current inventory,
safety stock, near-term sales, and inventory records.

Rules:

1. The backend supplies low-stock candidates from structured queries.
2. Near-term sales window defaults to last 7 days.
3. The LLM proposes replenishment reason, risk level, and suggested quantity
   based on the structured input.
4. The backend validates that suggested quantities are positive integers and
   referenced products exist in the input data.

Acceptance criteria:

1. Output includes product, current stock, safety stock, last 7-day sales,
   suggested quantity, reason, and risk level.
2. Products without sales data are marked with clear limitations instead of
   invented trends.
3. Products without safety stock are excluded or marked as not analyzable.
4. Model output that references unknown products, invalid quantities, or invalid
   risk levels is rejected and not saved.

### 5.3 Hot-Product Analysis

The LLM analyzes hot products for last 7 days and last 30 days using backend
sales aggregation and current inventory data.

Acceptance criteria:

1. Output includes product id, product name, sold quantity, order count, sales
   amount, current stock, safety stock, and inventory risk.
2. Risk level is generated by the LLM and validated against the allowed enum.
3. Time window can be `7` or `30` days.
4. The assistant explains why a hot product is or is not at inventory risk.

### 5.4 Replenishment Suggestion Generation

The LLM generates a replenishment suggestion list from low-stock, hot-product,
sales, and inventory-record evidence supplied by backend tools.

Suggestion item fields:

1. `productId`
2. `productName`
3. `availableStock`
4. `lockedStock`
5. `safetyStock`
6. `soldQuantityLast7Days`
7. `suggestedQuantity`
8. `reason`
9. `riskLevel`

Acceptance criteria:

1. Creating a suggestion does not change inventory.
2. Suggestion results are saved as `ai_operation_suggestion`.
3. Suggestion input snapshot is saved so the result can be reviewed later.
4. Model output is validated before persistence.
5. The persisted suggestion stores provider, model, prompt version, input
   snapshot, validated output, and validation status.
6. Risk level is enum-backed.

### 5.5 AI Suggestion Records

Suggestion statuses:

1. `DRAFT`
2. `CONFIRMED`
3. `REJECTED`
4. `EXECUTED`

Rules:

1. `DRAFT` is generated by AI analysis.
2. `REJECTED` means administrator rejected the suggestion.
3. `CONFIRMED` means administrator accepted the suggestion and converted it to
   an inbound order draft.
4. `EXECUTED` means the linked inbound order has been applied.
5. Suggestion records never directly mutate inventory.

Acceptance criteria:

1. Suggestions can be listed and viewed.
2. Suggestions can be rejected from `DRAFT`.
3. Suggestions can be confirmed from `DRAFT` to create an inbound order draft.
4. Rejected suggestions cannot create inbound orders.
5. Executed suggestions link to the final inbound order.

### 5.6 Inbound Order Draft Generation

The administrator can generate an inbound order draft based on an AI suggestion.

Rules:

1. Inbound order status starts as `DRAFT`.
2. Creating a draft does not modify inventory.
3. Draft source must link back to the AI suggestion number.

Acceptance criteria:

1. Draft contains product items and quantities from the confirmed suggestion.
2. Draft stores current stock, safety stock, and reason snapshot.
3. Draft can be reviewed before final confirmation.

### 5.7 Administrator Confirmation Execution

The administrator confirms an inbound order draft.

Rules:

1. Only `DRAFT` inbound orders can be confirmed.
2. Inventory-service applies stock increases.
3. Stock increase, inventory records, and operation logs are written in one
   transaction.
4. Duplicate confirmation must not increase inventory twice.

Acceptance criteria:

1. Successful confirmation updates inbound order status to applied/executed.
2. Inventory available stock increases by item quantities.
3. Inventory records contain before/after quantity, source type, source id, and
   operator id.
4. Admin operation log records the confirm action.
5. Repeating the same confirm request returns a deterministic result without a
   second stock increase.

### 5.8 Inventory Record Traceability

Every inventory change must expose:

1. `productId`
2. `changeType`
3. `quantity`
4. `beforeQuantity`
5. `afterQuantity`
6. `sourceType`
7. `sourceId`
8. `operatorId`
9. `createdAt`

Acceptance criteria:

1. Order deductions, order releases, admin adjustments, and inbound order
   confirmations all write traceable records.
2. AI suggestion links are visible through source or reference fields after an
   inbound order is created from a suggestion.

### 9.3 API Behavior

Inventory ask:

1. Accepts a question string and optional product id.
2. Uses an LLM intent-planning step constrained by a whitelist of allowed
   inventory operations.
3. Executes backend-owned logical tools for the matched intent.
4. Sends tool results to the LLM for analysis and natural-language response.
5. Returns answer text plus structured evidence.

Replenishment generation:

1. Reads low-stock products.
2. Reads last 7-day sales statistics.
3. Reads hot-product statistics and optional inventory records.
4. Sends structured evidence to the LLM.
5. The LLM generates suggested quantities, reasons, risk levels, and
   limitations.
6. The backend validates the model JSON output against the Phase 3 output
   schema.
7. Saves an `ai_operation_suggestion` only after validation passes.

Suggestion confirm:

1. Validates suggestion is `DRAFT`.
2. Creates inbound order draft.
3. Updates suggestion status to `CONFIRMED`.
4. Does not change inventory.

Inbound confirm:

1. Belongs to Phase 2.5 business APIs, not AI APIs.
2. Applies inventory change only after admin confirmation.

## 11. AI Prompt And Tool Design

### 11.1 Provider Strategy

Default provider:

1. `DEEPSEEK` with `deepseek-v4-flash`.

Enhanced analysis model:

1. `DEEPSEEK` with `deepseek-v4-pro`.

Optional provider:

1. `MINIMAX` with `MiniMax-M2.7`.

Fallback provider:

1. `MOCK`, used only for tests, local development without network, and demo
   fallback when external providers are unavailable.

Default configuration:

1. `AI_PROVIDER=deepseek`
2. `AI_MODEL=deepseek-v4-flash`
3. `AI_BASE_URL=https://api.deepseek.com`
4. `AI_MODEL_STRICT_JSON=true`
5. `AI_API_KEY` must come from environment variables or external secret
   configuration.

No API keys or secrets may be hardcoded. Browser code must never call model
providers directly.

### 11.2 System Prompt Rules

The system prompt must require:

1. Use only provided JSON data.
2. Do not invent product, order, sales, or inventory facts.
3. If data is missing, say the system lacks enough data.
4. Do not claim inventory was changed.
5. Do not output SQL.
6. Do not mention internal service endpoints.
7. Return structured JSON when the backend asks for structured output.

### 11.3 Tool Boundary

The LLM may request these logical tools only. The backend owns actual execution:

1. `getInventory(productId)`
2. `listLowStockProducts()`
3. `getSalesStats(days)`
4. `getHotProducts(days)`
5. `getInventoryRecords(productId)`
6. `getOperationSummary(days)`
7. `createSuggestionRecord(payload)`
8. `createInboundDraftFromSuggestion(suggestionNo)`

These logical tools must call application services or admin-safe APIs. They must
not execute SQL directly.

Tool execution rules:

1. Read tools can return only structured product, inventory, order, sales,
   inventory record, and operation summary data.
2. `createSuggestionRecord` can persist validated suggestions but cannot change
   inventory.
3. `createInboundDraftFromSuggestion` can create an inbound draft but cannot
   confirm it or change inventory.
4. Inventory changes remain exclusive to the Phase 2.5 inbound confirmation
   workflow.

### 11.4 LLM Analysis With Backend Guardrails

The LLM owns:

1. Administrator intent interpretation.
2. Inventory and sales analysis based on tool results.
3. Suggested replenishment quantity.
4. Risk level and reason.
5. Daily report prose.
6. Natural-language inventory answers.

The backend owns:

1. ADMIN authorization.
2. Logical tool execution.
3. Product and inventory existence checks.
4. Output schema validation.
5. Suggestion status transitions.
6. Inbound draft creation.
7. Formal inventory execution after administrator confirmation.

Backend validation rules:

1. Product ids in model output must exist in the structured input data.
2. `suggestedQuantity` must be a positive integer when present.
3. `riskLevel` must be one of `LOW`, `MEDIUM`, or `HIGH`.
4. The model must not claim inventory has already been changed.
5. The model must not output SQL.
6. The model must not cite products, sales values, stock values, or dates that
   are absent from the structured input.
7. Invalid model output is rejected and not saved as a confirmed suggestion.

### 11.5 Model Output Schema

The backend asks the model to return JSON compatible with this shape:

```json
{
  "summary": "string",
  "analysisType": "LOW_STOCK | HOT_PRODUCTS | DAILY_REPORT | INVENTORY_QA | REPLENISHMENT",
  "timeRange": {
    "from": "string",
    "to": "string"
  },
  "items": [
    {
      "productId": "string",
      "productName": "string",
      "availableStock": 0,
      "lockedStock": 0,
      "safetyStock": 0,
      "soldQuantityLast7Days": 0,
      "suggestedQuantity": 0,
      "riskLevel": "LOW | MEDIUM | HIGH",
      "reason": "string"
    }
  ],
  "limitations": ["string"]
}
```

Validation acceptance criteria:

1. JSON must parse successfully.
2. `analysisType` and `riskLevel` values must map to enums.
3. Every `productId` must come from the input snapshot.
4. Numeric fields must be non-negative.
5. Replenishment suggestions with `suggestedQuantity <= 0` are rejected unless
   the item is explicitly reported as a limitation rather than a suggestion.

## 16. Main Risks And Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Phase 2.5 prerequisites are incomplete | Phase 3 scope expands beyond two weeks | Treat Phase 2.5 entry gate as mandatory |
| AI hallucination | Wrong operational advice | Give the model only structured data, require JSON schema output, validate product ids, quantities, risk levels, and forbidden claims before persistence |
| Duplicate inbound execution | Inventory over-counting | Use inbound status, request id, transaction, and idempotency checks |
| Missing statistics data | Weak analysis quality | Add demo data generator and required aggregation APIs |
| Provider instability | Demo failure | Default implementation uses DeepSeek with `MOCK` fallback; MiniMax remains a provider adapter option |
| Frontend/backend contract drift | Integration delay | Keep Phase 3 APIs under one contract and add typed admin API clients |
| Overbuilding Agent behavior | Schedule failure | Use LLM-planned, backend-executed whitelist tools instead of autonomous multi-agent behavior |
