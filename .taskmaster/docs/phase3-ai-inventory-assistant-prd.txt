## 1. Overall Feasibility

Phase 3 is feasible for a two-week integrated training project only if the
Phase 2.5 readiness work is completed first.

Current repository capabilities:

1. Product, order, inventory, and inventory record base tables already exist.
2. Orders store `product_id`, `product_name`, `quantity`, `total_amount`,
   `status`, and timestamps, so basic 7-day and 30-day sales aggregation is
   possible.
3. Inventory supports current stock and locked stock.
4. Inventory deduct/release operations are already idempotent by order number
   and change type.
5. User roles and JWT role claims have started in Phase 2.

Current gaps:

1. Admin `/api/admin/**` route protection is not complete.
2. No finished admin inventory API for initialization, adjustment, low-stock
   query, or record timeline.
3. Existing `inventory_records` only tracks product, order number, change type,
   quantity, and status. It does not yet track before/after quantities, source,
   operator, request id, or reason.
4. No `admin_operation_logs` implementation is present yet.
5. No `inbound_order`, `inbound_order_item`, or `ai_operation_suggestion`
   tables are present yet.
6. No admin operation statistics APIs exist yet.
7. No independent `admin-frontend` implementation exists yet.

Conclusion:

Phase 3 should not directly implement stock mutation foundations. Those belong
to Phase 2.5. Phase 3 should focus on AI-facing analysis, suggestion generation,
review UI, and integration with the Phase 2.5 business workflow.

## 2. Product Goal

Build an AI Product Inventory Operations Assistant for MiniMall administrators.
The assistant uses structured product, order, inventory, inventory record, and
operation statistics data to provide:

1. Inventory Q&A.
2. Low-stock analysis.
3. Hot-product analysis.
4. Replenishment suggestion generation.
5. AI suggestion history and review.
6. Inbound order draft generation from AI suggestions.
7. Admin-reviewed inventory execution through the inventory service.
8. Daily inventory operations report.

The assistant must remain advisory. It must not directly update inventory.

## 3. Core Principles

1. AI does not directly modify inventory.
2. AI does not directly execute SQL.
3. AI does not bypass backend business services.
4. AI only generates analysis, operation suggestions, and document drafts.
5. After administrator confirmation, inventory-service executes the formal
   inventory change.
6. Every inventory change must write inventory records and admin operation
   logs.
7. AI responses must be based on structured system data and must not invent
   facts.
8. `/internal/**` remains blocked for browser and AI-facing callers.
9. All admin-facing APIs return `ApiResponse`.
10. Status fields use enums.

## 4. Phase 2.5 Entry Gate

Phase 3 implementation can start only after the following Phase 2.5 capabilities
exist:

1. Admin authentication and `/api/admin/**` authorization work.
2. Admin product list/detail APIs are available.
3. Admin inventory APIs can query current stock, low-stock products, and record
   timelines.
4. Inventory includes `safetyStock` or `lowStockThreshold`.
5. Inventory records include source, before/after quantity, operator, reason,
   request id, and business reference.
6. Admin operation logs exist.
7. Sales aggregation APIs exist for product-level 7-day and 30-day analysis.
8. Inbound order draft APIs exist.
9. Inbound order confirmation increases inventory transactionally, writes
   inventory records, writes admin operation logs, and prevents duplicate
   confirmation.
10. AI suggestion records exist as a business object, even if Phase 2.5 does
    not yet generate suggestions with a model.

If any entry-gate item is missing, implement it in Phase 2.5 rather than hiding
it inside Phase 3.

## 5. P0 Scope

### 5.1 Inventory Q&A And Query

The assistant supports these structured questions:

1. Query current inventory.
2. Query low-stock products.
3. Query a specific product's inventory status.

Acceptance criteria:

1. Answers cite the queried product, current stock, locked stock, safety stock,
   and stock state when available.
2. Unsupported questions return a controlled message, not a fabricated answer.
3. Answers include the data time range or query time.
4. The assistant uses only structured backend APIs.

### 5.2 Low-Stock Analysis

The assistant identifies products below safety stock and combines near-term
sales signals.

Rules:

1. Low-stock product: `availableStock < safetyStock`.
2. Near-term sales window: last 7 days.
3. Suggested replenishment quantity is calculated by backend rule, not by the
   language model.

Recommended minimum replenishment rule:

`max(safetyStock - availableStock, soldQuantityLast7Days)`

Acceptance criteria:

1. Output includes product, current stock, safety stock, last 7-day sales,
   suggested quantity, reason, and risk level.
2. Products without sales data still produce a deterministic suggestion based
   on safety stock gap.
3. Products without safety stock are excluded or marked as not analyzable.

### 5.3 Hot-Product Analysis

The assistant reports hot products for last 7 days and last 30 days.

Acceptance criteria:

1. Output includes product id, product name, sold quantity, order count, sales
   amount, current stock, safety stock, and inventory risk.
2. Risk level is computed by backend rule.
3. Time window can be `7` or `30` days.

### 5.4 Replenishment Suggestion Generation

The assistant generates a replenishment suggestion list from low-stock and
hot-product analysis.

Suggestion item fields:

1. `productId`
2. `productName`
3. `availableStock`
4. `lockedStock`
5. `safetyStock`
6. `soldQuantityLast7Days`
7. `suggestedQuantity`
8. `reason`
9. `riskLevel`

Acceptance criteria:

1. Creating a suggestion does not change inventory.
2. Suggestion results are saved as `ai_operation_suggestion`.
3. Suggestion input snapshot is saved so the result can be reviewed later.
4. Risk level is stable and enum-backed.

### 5.5 AI Suggestion Records

Suggestion statuses:

1. `DRAFT`
2. `CONFIRMED`
3. `REJECTED`
4. `EXECUTED`

Rules:

1. `DRAFT` is generated by AI analysis.
2. `REJECTED` means administrator rejected the suggestion.
3. `CONFIRMED` means administrator accepted the suggestion and converted it to
   an inbound order draft.
4. `EXECUTED` means the linked inbound order has been applied.
5. Suggestion records never directly mutate inventory.

Acceptance criteria:

1. Suggestions can be listed and viewed.
2. Suggestions can be rejected from `DRAFT`.
3. Suggestions can be confirmed from `DRAFT` to create an inbound order draft.
4. Rejected suggestions cannot create inbound orders.
5. Executed suggestions link to the final inbound order.

### 5.6 Inbound Order Draft Generation

The administrator can generate an inbound order draft based on an AI suggestion.

Rules:

1. Inbound order status starts as `DRAFT`.
2. Creating a draft does not modify inventory.
3. Draft source must link back to the AI suggestion number.

Acceptance criteria:

1. Draft contains product items and quantities from the confirmed suggestion.
2. Draft stores current stock, safety stock, and reason snapshot.
3. Draft can be reviewed before final confirmation.

### 5.7 Administrator Confirmation Execution

The administrator confirms an inbound order draft.

Rules:

1. Only `DRAFT` inbound orders can be confirmed.
2. Inventory-service applies stock increases.
3. Stock increase, inventory records, and operation logs are written in one
   transaction.
4. Duplicate confirmation must not increase inventory twice.

Acceptance criteria:

1. Successful confirmation updates inbound order status to applied/executed.
2. Inventory available stock increases by item quantities.
3. Inventory records contain before/after quantity, source type, source id, and
   operator id.
4. Admin operation log records the confirm action.
5. Repeating the same confirm request returns a deterministic result without a
   second stock increase.

### 5.8 Inventory Record Traceability

Every inventory change must expose:

1. `productId`
2. `changeType`
3. `quantity`
4. `beforeQuantity`
5. `afterQuantity`
6. `sourceType`
7. `sourceId`
8. `operatorId`
9. `createdAt`

Acceptance criteria:

1. Order deductions, order releases, admin adjustments, and inbound order
   confirmations all write traceable records.
2. AI suggestion links are visible through source or reference fields after an
   inbound order is created from a suggestion.

## 7. Non-Goals

Phase 3 does not include:

1. AI automatic outbound stock changes.
2. AI automatic product price changes.
3. AI automatic order cancellation.
4. AI automatic product deletion.
5. Multi-warehouse or multi-location inventory.
6. Supplier procurement system.
7. Complex forecasting models.
8. Complex multi-tool autonomous Agent behavior.
9. Admin RBAC beyond the existing `ADMIN` role unless Phase 2 adds it.
10. Replacing existing inventory-service business rules.

## 9. Backend API Recommendations

All APIs use `/api/admin/**` and return `ApiResponse`.

### 9.1 Phase 2.5 Required APIs

These must exist before Phase 3 starts:

1. `GET /api/admin/inventories`
2. `GET /api/admin/inventories/{productId}`
3. `GET /api/admin/inventories/low-stock`
4. `GET /api/admin/inventories/{productId}/records`
5. `GET /api/admin/operation-stats/sales-by-product?days=7`
6. `GET /api/admin/operation-stats/sales-by-product?days=30`
7. `GET /api/admin/operation-stats/inventory-trends`
8. `POST /api/admin/inbound-orders/drafts`
9. `GET /api/admin/inbound-orders`
10. `GET /api/admin/inbound-orders/{inboundNo}`
11. `POST /api/admin/inbound-orders/{inboundNo}/confirm`
12. `GET /api/admin/ai-suggestions`

### 9.2 Phase 3 AI APIs

1. `POST /api/admin/ai/inventory/ask`
2. `GET /api/admin/ai/inventory/low-stock-analysis?days=7`
3. `GET /api/admin/ai/inventory/hot-products?days=7`
4. `GET /api/admin/ai/inventory/hot-products?days=30`
5. `POST /api/admin/ai/replenishment-suggestions/generate`
6. `GET /api/admin/ai/suggestions`
7. `GET /api/admin/ai/suggestions/{suggestionNo}`
8. `POST /api/admin/ai/suggestions/{suggestionNo}/reject`
9. `POST /api/admin/ai/suggestions/{suggestionNo}/confirm`
10. `GET /api/admin/ai/reports/daily`

### 9.3 API Behavior

Inventory ask:

1. Accepts a question string and optional product id.
2. Maps question to a whitelist intent.
3. Runs structured data queries for the matched intent.
4. Returns answer text plus structured evidence.

Replenishment generation:

1. Reads low-stock products.
2. Reads last 7-day sales statistics.
3. Computes suggested replenishment quantity with backend rules.
4. Optionally asks the model to generate human-readable reasons from the
   structured result.
5. Saves an `ai_operation_suggestion`.

Suggestion confirm:

1. Validates suggestion is `DRAFT`.
2. Creates inbound order draft.
3. Updates suggestion status to `CONFIRMED`.
4. Does not change inventory.

Inbound confirm:

1. Belongs to Phase 2.5 business APIs, not AI APIs.
2. Applies inventory change only after admin confirmation.

## 11. AI Prompt And Tool Design

### 11.1 Provider Strategy

Support two providers:

1. `MOCK`: deterministic provider for tests and demo fallback.
2. `OPENAI` or external provider: enabled only through environment
   configuration.

No API keys or secrets may be hardcoded.

### 11.2 System Prompt Rules

The system prompt must require:

1. Use only provided JSON data.
2. Do not invent product, order, sales, or inventory facts.
3. If data is missing, say the system lacks enough data.
4. Do not claim inventory was changed.
5. Do not output SQL.
6. Do not mention internal service endpoints.
7. Return structured JSON when the backend asks for structured output.

### 11.3 Tool Boundary

The AI orchestration layer may use these logical tools only:

1. `getInventory(productId)`
2. `listLowStockProducts()`
3. `getSalesStats(days)`
4. `getHotProducts(days)`
5. `getInventoryRecords(productId)`
6. `createSuggestionRecord(payload)`
7. `createInboundDraftFromSuggestion(suggestionNo)`

These logical tools must call application services or admin-safe APIs. They must
not execute SQL directly.

### 11.4 Deterministic Computation

The backend, not the model, owns:

1. Low-stock classification.
2. Hot-product ranking.
3. Suggested replenishment quantity.
4. Risk level.
5. Suggestion status transitions.

The model may generate:

1. Human-readable explanation.
2. Daily report prose.
3. Natural-language inventory answer based on structured evidence.

## 16. Main Risks And Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Phase 2.5 prerequisites are incomplete | Phase 3 scope expands beyond two weeks | Treat Phase 2.5 entry gate as mandatory |
| AI hallucination | Wrong operational advice | Backend computes numeric suggestions; model only explains structured data |
| Duplicate inbound execution | Inventory over-counting | Use inbound status, request id, transaction, and idempotency checks |
| Missing statistics data | Weak analysis quality | Add demo data generator and required aggregation APIs |
| Provider instability | Demo failure | Default to `MOCK` provider; real provider is optional |
| Frontend/backend contract drift | Integration delay | Keep Phase 3 APIs under one contract and add typed admin API clients |
| Overbuilding Agent behavior | Schedule failure | Avoid multi-tool autonomous Agent; use deterministic workflows |
