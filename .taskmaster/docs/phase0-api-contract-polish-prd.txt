版本：

当前说明：
本 PRD 是已完成的 Phase 0 API Contract Polish 历史基线，负责把早期
gateway/service-prefix 路径收敛到 canonical browser API。Phase 2 后台、
Phase 2.5 AI 库存准备、Phase 3 AI 库存助手的当前接口现状，以对应阶段
合同和验收文档为边界背景；当前唯一浏览器/API 接口总览以
docs/api-gateway-contract.md 为准。不要把本文中的“未来/暂不定义 admin
API”表述当作当前项目状态。

目标阶段：
Phase 0 - API Contract Polish

适用项目：
MiniMall Order 微服务电商后端项目


一、背景

MiniMall Order 后端 MVP 已完成第一轮 20 个 TaskMaster 顶层任务，并通过本地后端验收。当前系统已经具备：

1. Java 17 + Spring Boot 3 + Spring Cloud Maven 多模块工程。
2. api-gateway 统一浏览器入口。
3. user-service、product-service、inventory-service、order-service、payment-service、notification-service 等微服务。
4. MySQL、Redis、RabbitMQ、Prometheus、Grafana 本地运行环境。
5. 注册、登录、当前用户、商品查询、库存查询、创建订单、取消订单、模拟支付、支付成功事件消费、通知日志等核心链路。
6. 统一 ApiResponse 响应结构。
7. BusinessException + ErrorCode 统一业务错误处理。
8. JWT 鉴权和用户上下文透传。
9. RabbitMQ 支付成功事件。
10. k6 压测脚本和最终验收文档。

当前项目已经可以被视为“本地后端 MVP”，但交付范围仍是 backend only，不包含用户前端页面、管理员后台、RBAC、退款、对账、库存管理操作、通知管理 API、生产 CI/CD 等能力。

在进入用户前端和管理员后台开发之前，需要先完成一次 API Contract Polish，避免后续前端绑定到不够稳定、不够清晰或带有临时痕迹的 API 路径和 DTO 字段上。


二、当前主要问题

2.1 Gateway 外部路径不够清爽

当前浏览器入口通过 api-gateway 暴露的路径类似：

- POST /api/user/users/register
- POST /api/user/users/login
- GET /api/product/products
- GET /api/order/orders/{orderNo}
- POST /api/payment/payments/{orderNo}/pay

这些路径功能可用，但存在双层资源名：

- /api/user/users
- /api/product/products
- /api/order/orders
- /api/payment/payments

对未来前端而言，这类路径不够直观。更适合前端接入的 canonical API 应为：

- /api/users
- /api/products
- /api/inventories
- /api/orders
- /api/payments


2.2 Product admin endpoints 已存在，但没有 RBAC

当前商品创建、更新、上下架接口已经存在，但没有管理员角色和权限控制。因此这些接口不能被视为真正的“管理员后台 API”。

Phase 0 不实现 RBAC，也不实现管理员后台，但需要避免在用户前端文档中误导性地把商品写操作当作普通用户 API。


2.3 /internal/** 必须继续对浏览器不可见

当前后端已有服务间内部接口，例如商品内部查询、库存扣减、库存释放等。Phase 0 必须继续确保：

1. 浏览器不能访问 /internal/**。
2. api-gateway 不转发 /internal/**。
3. 前端文档不出现 /internal/** 作为可调用接口。
4. 压测脚本不直接调用 /internal/**。


2.4 API 文档、压测脚本、前端接入文档需要统一

现有后端已经有：

- docs/frontend-integration.md
- docs/api-gateway-contract.md
- docs/api-contract-review.md
- pressure/README.md
- pressure/task19-live-result.md

Phase 0 需要将这些文档中的外部 API 路径统一到最终规范，避免用户前端 PRD 和后续管理员后台 PRD 继续沿用旧路径。


2.5 Product status mutation 风格需要固定

当前商品上下架接口为：

- POST /api/product/products/{productId}/on-shelf
- POST /api/product/products/{productId}/off-shelf

未来管理员后台更适合使用统一状态更新接口，例如：

- PUT /api/admin/products/{productId}/status

Phase 0 不必实现完整后台权限，但应明确未来商品状态接口采用哪种设计，以免管理员后台开发时出现双口径。


2.6 PageResponse<T> 可能需要共享

当前分页响应如果只在 product-service 内部使用，可以暂不迁移；但订单列表、后台商品列表、库存列表、通知日志列表都需要分页时，应统一到 common-core，避免各服务复制不同分页结构。


三、目标

Phase 0 的目标是：

1. 明确 MiniMall 对外浏览器 API 的最终路径规范。
2. 让用户前端只依赖清晰、稳定、复数资源风格的 /api/** 路径。
3. 保持 /internal/** 不暴露给浏览器。
4. 避免前端误用服务内部 API、数据库主键或旧 gateway prefix。
5. 修正或标记 DTO 命名、分页结构、商品状态接口风格等契约问题。
6. 更新文档、压测脚本和 E2E 验收脚本，使它们使用新的 canonical API。
7. 不引入前端 UI。
8. 不实现管理员后台。
9. 不实现 RBAC。
10. 不扩展支付、订单、通知等业务生命周期。


四、非目标

Phase 0 明确不做以下事情：

1. 不开发用户前端页面。
2. 不开发管理员后台页面。
3. 不实现 RBAC、角色、权限、管理员账号体系。
4. 不实现库存管理后台 API。
5. 不实现退款、对账、支付回调、支付渠道抽象。
6. 不实现订单发货、履约、售后、退款状态。
7. 不实现通知模板、重发、失败告警。
8. 不做高并发架构改造。
9. 不做 Kubernetes、CI/CD、镜像发布、Secrets 管理。
10. 不重构已有核心业务逻辑。
11. 不修改已经通过验收的业务语义。

六、目标 API 路径规范

6.1 Public Customer API

用户相关：

1. POST /api/users/register
   说明：用户注册。

2. POST /api/users/login
   说明：用户登录。

3. GET /api/users/me
   说明：当前用户。

商品相关：

1. GET /api/products
   说明：商品列表。

2. GET /api/products/{productId}
   说明：商品详情。

库存相关：

1. GET /api/inventories/{productId}
   说明：商品库存快照。

订单相关：

1. POST /api/orders
   说明：创建订单。

2. GET /api/orders/my
   说明：我的订单。

3. GET /api/orders/{orderNo}
   说明：订单详情。

4. POST /api/orders/{orderNo}/cancel
   说明：取消订单。

支付相关：

1. POST /api/payments/{orderNo}/pay
   说明：模拟支付。

2. GET /api/payments/{orderNo}
   说明：查询支付单。


6.2 Internal API

内部 API 继续使用：

1. /internal/products/**
   使用方：order-service。

2. /internal/inventories/**
   使用方：order-service。

3. /internal/orders/**
   使用方：payment-service，如已有或后续需要。

要求：

1. api-gateway 不暴露 /internal/**。
2. 前端文档不出现 /internal/**。
3. 压测脚本不调用 /internal/**。
4. internal DTO 不作为前端 DTO 文档的一部分。


6.3 Admin API 暂不正式定义

Phase 0 不正式定义 /api/admin/** 的完整接口。

但文档中必须标记：

1. 当前商品写接口不是完整 admin-safe API。
2. 后续 Phase 2 应定义 /api/admin/products/**。
3. 后续 Phase 2 应实现 RBAC 后再暴露库存调整、订单管理、通知日志管理等后台功能。


七、Gateway 路由要求

7.1 Canonical 路由

api-gateway 必须支持 canonical paths，并重写到下游服务现有 Controller。

建议路由映射：

1. /api/users/**
   下游：user-service /api/users/**。

2. /api/products/**
   下游：product-service /api/products/**。

3. /api/inventories/**
   下游：inventory-service /api/inventories/**。

4. /api/orders/**
   下游：order-service /api/orders/**。

5. /api/payments/**
   下游：payment-service /api/payments/**。

如果下游 Controller 已经使用相同路径，则尽量减少 RewritePath 复杂度。


7.2 Legacy 路由策略

当前旧路径包括：

1. /api/user/users/**
2. /api/product/products/**
3. /api/inventory/inventories/**
4. /api/order/orders/**
5. /api/payment/payments/**

Phase 0 必须二选一：

方案 A：直接废弃 legacy 路由

1. Gateway 不再暴露旧路径。
2. 文档、压测、验收全部改成 canonical paths。
3. 本地 MVP 没有真实前端依赖旧路径，因此可以接受。

方案 B：短期保留 legacy alias

1. Legacy paths 继续可用。
2. 文档标记 deprecated。
3. 所有新文档、压测、前端示例只使用 canonical paths。
4. 后续 Phase 1 前端禁止使用 legacy paths。

优先推荐方案 A。若改动风险过大，可以选择方案 B，但必须在文档中明确 deprecated。


7.3 Gateway 安全要求

Gateway 必须继续满足：

1. 浏览器请求必须通过 Authorization: Bearer <jwt> 访问受保护接口。
2. 注册、登录、CORS preflight 可放行。
3. 浏览器传入的 X-User-Id、X-Username 必须被移除。
4. JWT 校验通过后由 Gateway 注入可信 X-User-Id、X-Username。
5. /internal/** 不得被 Gateway 转发。
6. Gateway 产生的 401、403、404、429、500 错误必须返回 ApiResponse。


八、DTO 与响应规范

8.1 统一响应

所有 public API 成功响应必须使用：

{
  "success": true,
  "code": "0",
  "message": "success",
  "data": {}
}

所有 public API 失败响应必须使用：

{
  "success": false,
  "code": "40100",
  "message": "Unauthorized",
  "data": null
}

具体 code 和 message 根据 ErrorCode 决定。


8.2 外部 ID 规则

前端稳定标识必须使用业务 ID：

1. 用户：userId。
2. 商品：productId。
3. 订单：orderNo。
4. 支付：paymentNo。
5. 事件：eventId，仅内部或文档需要。

禁止在 public frontend DTO 中暴露裸 id 数据库主键。


8.3 分页结构

如果 Phase 0 发现多个服务已需要分页，必须迁移或准备迁移：

common-core/src/main/java/com/minimall/common/core/response/PageResponse.java

统一结构：

{
  "content": [],
  "page": 0,
  "size": 10,
  "totalElements": 100,
  "totalPages": 10
}

如果当前只有 product-service 使用分页，可以将迁移作为 P1 或 P2，但必须在文档中记录后续 order/admin 分页需要复用该结构。


8.4 Request / Response 命名

规则：

1. 请求 DTO：*Request。
2. 响应 DTO：*Response。
3. 分页包装：PageResponse<T>。
4. 事件：*Event。
5. 内部服务调用响应：可使用 Internal*Response 或 *Snapshot。

禁止新增：

1. *VO。
2. *Result。
3. *InfoDTO。
4. *ViewObject。
5. 与现有风格不一致的 DTO 命名。


九、商品状态接口决策

Phase 0 必须明确商品上下架未来接口风格。

推荐方案：

未来管理员后台使用统一状态更新接口：

PUT /api/admin/products/{productId}/status

请求体：

{
  "status": "ON_SHELF"
}

可选状态：

1. ON_SHELF。
2. OFF_SHELF。

当前兼容：

当前已有：

1. POST /api/product/products/{productId}/on-shelf
2. POST /api/product/products/{productId}/off-shelf

Phase 0 不强制删除已有后端接口，但必须在文档中明确：

1. 当前 on/off-shelf 是后端已有能力。
2. 未来管理员后台优先使用统一 status mutation。
3. 若 Phase 0 顺手改成统一接口，必须保证现有测试、文档、压测同步更新。
4. 不得在没有 RBAC 的情况下把商品写操作作为 customer frontend API 暴露。
